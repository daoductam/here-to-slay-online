# 1. Context

**Problem:**
The game is currently hardcoded to start only when there are at least 3 players and all are ready. We want to support customizable lobby sizes from 2 to 5 players, and enforce that the game starts only when the lobby is fully populated and everyone is ready.

**Affected Modules:**
- `server/src/types.ts` — Room & GameState types update.
- `server/src/controllers/express/homeController.ts` — Create & Join room APIs to handle/restrict based on target player count.
- `server/src/controllers/socketio/lobby/lobby.ts` — Socket.io logic for ready state and match start trigger.
- `client/src/types.ts` — GameState types sync.
- `client/src/App.tsx` — Add UI dropdown for player count and display player limits in Room finder.
- `client/src/pages/Lobby.tsx` — Adjust UI placeholder slots dynamically.

**Non-Goals:**
- Allowing players to change the target player limit after room creation.
- Spectating features or exceeding 5 players.

---

# 2. Constraints

**Language:** TypeScript
**Architecture:** Node Express/Socket.io backend + React frontend

**Rules:**
- Do NOT introduce new external dependencies.
- Maintain backward compatibility (fall back to 3 players if a room is loaded without `targetPlayers`).

# 4. Contracts

## Data Structure: `GameState['match']`

```typescript
match: {
  gameStarted: boolean;
  players: string[];
  isReady: (boolean | null)[];
  startRolls: { maxVal: number; inList: number[]; rolls: number[] };
  paused?: boolean;
  disconnectTimeLeft?: number;
  disconnectedPlayerNum?: number | null;
  targetPlayers?: number; // Target number of players (2 to 5). Defaults to 3.
}
```

## DTO changes: `/create-room` Request

```json
{
  "roomId": "string",
  "isPrivate": "boolean",
  "username": "string",
  "targetPlayers": "number" // Optional, 2-5, defaults to 3
}
```

## DTO changes: `/get-rooms` Response

```json
{
  "roomId": {
    "joined": "number",
    "target": "number"
  }
}
```

# 5. Target Architecture

**Components:**
- **Express router**: `/create-room` receives `targetPlayers` and registers it in `rooms[roomId]`. `/get-rooms` returns an object of `{ [id: string]: { joined: number, target: number } }` or keeps the record updated. Wait! To prevent breaking clients, we can keep the value in `/get-rooms` as a number if we want to minimize client-side modifications, or change it. Let's see: `client/src/App.tsx` expects `rooms` to be `{ [key: string]: number }` (joined player count).
  Wait! Let's check `App.tsx` lines 183-194:
  ```typescript
  {Object.keys(rooms).map(id => (
    <div className='room' key={id} onClick={() => joinRoom(id)}>
      <div>{id}</div>
      <div>Players: {rooms[id]}</div>
    </div>
  ))}
  ```
  If we change `/get-rooms` response type, we might break existing clients. However, if we change `rooms[id]` response to be an object or we can return a formatted string or keep it simple. But wait! If we return an object, we can easily type it on both sides.
  Actually, if we return `rooms` as `{ [key: string]: { joined: number, target: number } }`, let's update both server and client to use this new structure. It's clean and type-safe.

**Interaction Flow:**
```
Creator Browser -> POST /create-room { roomId, isPrivate, username, targetPlayers } -> server sets rooms[roomId].state.match.targetPlayers = targetPlayers
Client Browser -> GET /get-rooms -> server returns { [roomId]: { joined: numPlayers, target: targetPlayers } }
Joiner Browser -> POST /join-room -> server checks room.numPlayers < targetPlayers. If full, returns 400 "Room is full".
Lobby -> Socket connection -> when all players are ready and numPlayers === targetPlayers -> trigger start-match
```

# 8. Task Specifications

## TASK-001: Update Shared Types
- **Files**:
  - `server/src/types.ts` — **modify**: Add `targetPlayers?: number` to `match` field in `State` interface.
  - `client/src/types.ts` — **modify**: Add `targetPlayers?: number` to `match` field in `GameState` interface.
- **Acceptance Criteria**:
  - Code compiles without type errors.

## TASK-002: Update Express Room API & Initial State
- **Files**:
  - `server/src/cards.ts` — **modify**: Set `targetPlayers: 3` as default in `initialState.match`.
  - `server/src/controllers/express/homeController.ts` — **modify**:
    - Update `getRooms` to return `{ [key: string]: { joined: number, target: number } }`.
    - Update `createRoom` to read `targetPlayers` from body (validate range `2 <= targetPlayers <= 5`, default 3) and set `rooms[roomId].state.match.targetPlayers = targetPlayers`.
    - Update `joinRoom` to check if `rooms[roomId].numPlayers >= targetPlayers` and if so, return status `400` with `Room is full` message.
- **Acceptance Criteria**:
  - `/create-room` successfully saves the custom target player count.
  - `/join-room` rejects attempts when the room is full.

## TASK-003: Update Socket Lobby Match-Start Checks
- **Files**:
  - `server/src/controllers/socketio/lobby/lobby.ts` — **modify**:
    - In `ready`, change the check for match start from `>= 3` to `=== targetPlayers` (defaulting to 3 if undefined).
    - In `startMatch`, change the check for match start from `>= 3` to `=== targetPlayers` (defaulting to 3 if undefined).
- **Acceptance Criteria**:
  - Match starts automatically only when exactly `targetPlayers` are ready.

## TASK-004: Update React Create Room & Find Rooms UI
- **Files**:
  - `client/src/App.tsx` — **modify**:
    - Update `rooms` state type to `{[key: string]: { joined: number, target: number }}`.
    - Add UI selector for player count (2 to 5) when creating a room.
    - Send selected `targetPlayers` when creating a room.
    - Display `Players: {rooms[id].joined}/{rooms[id].target}` in find rooms list.
- **Acceptance Criteria**:
  - User can select player count 2-5.
  - Finder list shows current/max player limits correctly.

## TASK-005: Update React Lobby Slots UI
- **Files**:
  - `client/src/pages/Lobby.tsx` — **modify**:
    - Calculate blank slots dynamically: `(matchState.targetPlayers || 3) - matchState.players.length`.
- **Acceptance Criteria**:
  - Shows the correct number of loading placeholder circles for the chosen player size.

# 9. Edge Cases

| # | Scenario | Expected Behavior | Handled In |
|---|----------|-------------------|------------|
| 1 | Creator enters an invalid player count (e.g. 1 or 6) | Backend validates and returns error or defaults to 3 | TASK-002 |
| 2 | Room reaches target count | `/join-room` prevents further players from joining | TASK-002 |
| 3 | Lobby state targetPlayers is missing (older games/fallback) | Defaults to 3 players | TASK-002, TASK-003, TASK-005 |

# 10. Risks

- **Risk**: Desynchronization between `rooms[roomId].numPlayers` and `rooms[roomId].state.match.players.length`.
  - **Mitigation**: Ensure all comparisons use the backend state's source of truth.
