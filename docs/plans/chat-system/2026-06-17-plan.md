# 1. Context

**Problem:**
Players inside rooms need a way to communicate textually in real-time during both the Lobby phase (lobby rooms) and active Matches (gameboard interface). Reconnecting or transitioning between pages should not clear the room's message history.

**Affected Modules:**
- `client/src/types.ts` — Type definitions for chat messages.
- `server/src/types.ts` — Type definitions matching client chat messages.
- `server/src/rooms.ts` — Room state representation containing chat history.
- `server/src/server.ts` — Registering the socket-chat controller.
- `server/src/controllers/socketio/chat.ts` (NEW) — Socket action handlers for chat.
- `client/src/components/ChatSidebar.tsx` (NEW) — Floating side-panel UI for the chat interface.
- `client/src/style/chat.css` (NEW) — Dark-fantasy themed chat styling.
- `client/src/pages/Lobby.tsx` — Integrate ChatSidebar component.
- `client/src/pages/Game.tsx` — Integrate ChatSidebar component and play sound cue.
- `client/src/components/MenuButtons.tsx` — Add Toggle Chat button and notification badge.

**Non-Goals:**
- Private messaging (Whisper/DM).
- Chat filters/moderation rules.
- Persistent database storage of chat messages (chats are cleared when room is closed/deleted).

# 4. Contracts

## Shared DTOs:

```typescript
// Add to both client/src/types.ts and server/src/types.ts
export interface ChatMessage {
  id: string;        // Unique message ID
  sender: string;    // Username of player (or "Hệ thống" for server status messages)
  text: string;      // Content of the message
  timestamp: number; // Unix timestamp
  isSystem?: boolean; // True if it is a system-generated event message
}
```

## Socket Events:

```
socket.emit('chat:send', roomId: string, userId: string, text: string)
  - pre: text is non-blank; roomId and userId represent a valid room player session
  - post: Broadcasts `chat:message` to all clients in the room, adds message to room.chatHistory

socket.on('chat:message', (message: ChatMessage) => void)
  - post: Client appends the ChatMessage to local active list, triggers notification sound if appropriate

socket.on('chat:history', (history: ChatMessage[]) => void)
  - post: Client sets its initial chat history to `history` list on join/reconnect
```

# 5. Target Architecture

**Components:**
- `ChatSidebar` — Manages state for chat messages list, inputs, submit events, auto-scroll to bottom.
- `socketio/chat` — Server handler to parse incoming messages, construct `ChatMessage` object, store in memory, and broadcast.

**Interaction Flow:**

```
Client (User submits message in ChatSidebar)
  → socket.emit('chat:send', roomId, userId, text)
    → Server handler reads sender username using checkCredentials
    → Creates ChatMessage object
    → Appends to rooms[roomId].chatHistory (slice to keep max 100)
    → io.in(roomId).emit('chat:message', message)
  ← Client listens to 'chat:message' -> updates UI state -> plays sound
```

# 8. Task Specifications

## TASK-001: Implement Server-Side Socket Chat Logic

**Description:**
Define type models for `ChatMessage` inside server types, add `chatHistory` storage inside room objects, and write the socket event listener for `chat:send`. Register the socket controller inside `server.ts`.

**Input:** None
**Output:** Chat socket endpoints and server-side state.
**Files:**
- `server/src/types.ts` — **modify**: Add `ChatMessage` interface. Update `Room` state interface to hold `chatHistory: ChatMessage[]`.
- `server/src/rooms.ts` — **modify**: Initialize `chatHistory: []` when creating new rooms.
- `server/src/controllers/socketio/chat.ts` — **create**: Handle `chat:send` socket event. Check credentials, format message, update history, broadcast.
- `server/src/server.ts` — **modify**: Register `chat:send` listener. Also, emit `chat:history` inside `enter-lobby` or startup socket connections.

**Acceptance Criteria:**
- Server code builds successfully (`npx tsc` executes without warning/error).
- When a user sends a message, server saves it in memory and broadcasts to the room.
- Max messages in room history is capped at 100.

# 9. Edge Cases

| # | Scenario | Expected Behavior | Handled In |
|---|----------|-------------------|------------|
| 1 | Empty message text | Do not send empty or purely whitespace messages | TASK-002 |
| 2 | Long message overflows bubble | Words wrap cleanly, no layout breaks | TASK-002 |
| 3 | Transitions from Lobby to Game | Chat history remains intact because Server holds history and resends on re-join | TASK-003 |
| 4 | Client disconnects and re-joins | Server sends full `chatHistory` on lobby rejoin to restore window state | TASK-001 |
| 5 | Sound cue on user's own message | Do NOT play notification sound when user themselves send a message | TASK-003 |
